# Code Review 自查清单

每次创建 PR 或更新 PR 前，必须按本文件执行自查 Code Review。此文件是 AGENTS.md 引用的强制 CR 要求。

## 1. 本地改动隔离检查

- [ ] 已执行 `git status --short`。
- [ ] 已确认没有误混入用户原有未提交改动。
- [ ] 如存在用户原有改动，已在用户确认后 stash / commit / 保留，并在回复中说明。
- [ ] 没有把无关依赖升级、构建产物、缓存文件混入本次 PR。

## 2. API / 文档一致性检查

- [ ] 新增、删除或修改公开 API 时，已同步更新 README / docs / 类型声明。
- [ ] README 中的函数签名、参数、事件、slot、ref/expose 与源码一致。
- [ ] 删除文档中的已有 API 说明时，已确认源码不存在该能力或文档原本错误。
- [ ] 若删除或改写已有文档承诺，已在回复或 PR 描述中说明原因。

## 3. 注释保留检查

- [ ] 未擅自删除已有且有用的注释。
- [ ] 重构时已迁移仍然有效的注释。
- [ ] 删除失效、误导或空逻辑注释时，已说明原因。
- [ ] 清理调试日志时，已确认不是有用注释或用户需要保留的行为。

## 4. 单测中文检查

- [ ] 新增测试文件中的 `describe` / `it` 描述使用中文。
- [ ] 修改测试时，没有无故把中文描述改成英文。
- [ ] 测试名称能表达业务行为，而不是只描述实现细节。

## 5. 验证检查

根据改动范围选择验证命令，并在 PR 描述中列出实际执行结果。

- [ ] 纯 Markdown / 规范文档变更：至少执行 `git diff --check`。
- [ ] 源码或类型变更：执行 `pnpm lint`、`pnpm typecheck`。
- [ ] core 行为变更：执行 `pnpm --filter @seamless-scroll/core test -- --run`。
- [ ] 构建或文档站变更：执行 `pnpm build`、`pnpm docs:build`。
- [ ] 如命令有 warning 但成功，PR 描述中必须说明 warning 内容。

## 6. PR 描述检查

- [ ] Summary 清楚说明用户可感知变化。
- [ ] Validation 列出实际执行过的命令。
- [ ] Notes 说明已知 warning、未覆盖项、用户 stash 或后续注意事项。
- [ ] PR 不包含与当前任务无关的改动。
